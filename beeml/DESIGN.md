# ASR Correction Training Pipeline вҖ” Design

## Goal

Train a small local model that corrects ASR (speech-to-text) errors, especially for domain-specific vocabulary like library names, CLI flags, and technical jargon.

### Training

- **Base model**: Qwen2.5-0.5B (smallest available, ~1.3GB peak memory)
- **Method**: LoRA fine-tune via MLX-LM
- **Format**: Completions вҖ” `prompt вҶ’ completion`
- **Prompt format**: `<vocab> term1 term2 <parakeet> noisy A <qwen> noisy B <correct>`
- **Completion**: ` corrected text<|endoftext|>`

### Written vs Spoken Forms

Every vocab term has two representations:
- **Written**: `SHA-256`, `--doc`, `Cargo.toml`, `serde` (correction target)
- **Spoken**: `shaw two fifty six`, `dash dash doc`, `cargo dot toml`, `sir dee` (fed to TTS)

The TTS synthesizes the spoken form. The correction model learns to recover the written form.

## Eval Strategy

### The eval that matters

**Your voice вҶ’ ASR(s) вҶ’ corrector вҶ’ final text**

Synthetic eval tells you if the pipeline works. Real-voice eval tells you if the product works.

### What to store per eval sentence

- Raw audio (WAV, 16kHz mono)
- Raw Parakeet transcript
- Raw Qwen3 transcript
- Corrector output
- Ground truth text
- Exact model/version/quantization for each component
- Per-sentence tags

### Per-sentence tags

- `rare-term` вҖ” domain-specific vocabulary
- `product-name` вҖ” proper nouns (HuggingFace, Xcode)
- `path` вҖ” file paths, URLs
- `code` вҖ” code-like syntax (Vec<T>, impl Trait)
- `number-version` вҖ” versions, numbers (SHA-256, v2.1)
- `easy-english` вҖ” common words, should be unchanged
- `ambiguous` вҖ” pronunciation unclear

### Key metrics

1. **Exact recovery of important terms** вҖ” did the corrector restore the vocab term?
2. **Improvement over best raw ASR** вҖ” did the final output beat the better of the two raw transcripts?
3. **Overcorrection rate** вҖ” on already-correct text, how often does the corrector change anything?

The single most important sanity metric:

> **"% of sentences where the corrector made the best available transcript worse."**

This catches the dangerous failure mode early.

### Mic profiles

Eval recordings are organized by mic profile to test across setups:
- `desk-sm7b` вҖ” studio mic at desk
- `laptop-bed` вҖ” laptop mic, casual
- etc.

Same sentences recorded with different mics = test acoustic robustness.

## Training Data Composition

### Correction examples (~70%)
Generated by the pipeline: TTS вҶ’ dual ASR вҶ’ JSONL.
The model learns to fix ASR errors using vocab context.

### Identity/unchanged examples (~20%)
Both ASR outputs are set to the correct text.
The model learns: **don't touch text that's already correct.**
No TTS or ASR needed вҖ” generated mechanically from vocab.

### Hard negatives (~10%)
Weird-looking terms that ARE correct and should NOT be changed:
- `GGUF`, `Blake3`, `wgpu`, `ratatui`
- File paths, crate names, CLI flags
- Things that look like errors but aren't

These teach restraint. The desired behavior is:

> Copy almost everything, intervene rarely, and be very sure when intervening.

## Future Improvements

- **LLM-assisted text generation** вҖ” more natural sentences than templates
- **Human-curated vocab** вҖ” review and fix pronunciation overrides
- **Multiple TTS voices** вҖ” speaker diversity to prevent overfitting
- **Real accepted corrections** вҖ” log user corrections as training signal
- **Seq2seq model** вҖ” if candidate ranking isn't enough
- **Integration into hark** вҖ” run corrector as post-processing after ASR
